## Background / Problem Statement

AI-powered applications built on Prefab frequently need **semantic similarity
search** — finding records that are conceptually related to a query, not just
lexically matching. The standard approach is to store vector embeddings alongside
record data and query using cosine (or dot-product) distance.

PostgreSQL's `pgvector` extension provides this natively, but integrating it with
Prefab's Spring Data JDBC-backed aggregates currently requires:

1. Manually writing the Flyway migration to `CREATE EXTENSION vector` and add a
   `vector(N)` column — outside Prefab's code-generation model.
2. Registering a custom Spring Data JDBC `Converter` pair for `float[]` ↔
   `PGvector`.
3. Writing a `@RepositoryMixin` with a raw `@Query` using the `<=>` operator.
4. Manually adding the IVFFlat index in a separate migration file.

This is ~80 lines of boilerplate per embedding field. The Maestro project
(`MemoryEntry` aggregate) reproduces this pattern today; any Prefab application
adding semantic search will reproduce it again.

A new `prefab-vector` module would make embedding fields a first-class Prefab
concept, reducing the above to a single annotation.

## Proposed API

### New module: `prefab-vector`

**Artifact ID:** `prefab-vector`
**Depends on:** `prefab-core`, `prefab-postgres`
**Optional dependency:** `pgvector` JDBC driver (`com.pgvector:pgvector`)

```xml
<dependency>
    <groupId>be.appify.prefab</groupId>
    <artifactId>prefab-vector</artifactId>
    <version>LATEST_VERSION</version>
</dependency>
```

### New annotation: `@Embedding`

**Package:** `be.appify.prefab.vector.annotations`
**Target:** `FIELD`
**Retention:** `RUNTIME`

```java
@Aggregate
@GetById
@GetList
public record MemoryEntry(
        @Id Reference<MemoryEntry> id,
        @Version long version,
        Reference<ConversationSession> session,
        MemoryType memoryType,
        @Text String content,
        @Embedding(dimensions = 1536) float[] embedding
) {
    @Create
    public MemoryEntry(Reference<ConversationSession> session,
                       MemoryType memoryType,
                       String content,
                       float[] embedding) {
        this(Reference.create(), 0L, session, memoryType, content, embedding);
    }
}
```

| Attribute | Type | Default | Description |
|-----------|------|---------|-------------|
| `dimensions` | `int` | — **(required)** | Number of dimensions in the embedding vector. Must match the model output (e.g. 1536 for `text-embedding-3-small`, 768 for `nomic-embed-text`). |
| `distanceFunction` | `DistanceFunction` | `COSINE` | Distance function for similarity queries: `COSINE` (`<=>`), `L2` (`<->`), `INNER_PRODUCT` (`<#>`). |
| `indexType` | `IndexType` | `IVFFLAT` | Vector index type: `IVFFLAT` or `HNSW`. |
| `indexLists` | `int` | `100` | IVFFlat `lists` parameter (ignored for HNSW). |
| `indexM` | `int` | `16` | HNSW `m` parameter (ignored for IVFFlat). |
| `indexEfConstruction` | `int` | `64` | HNSW `ef_construction` parameter (ignored for IVFFlat). |

### Generated artefacts

#### 1. Flyway migration additions (appended to the aggregate's generated script)

```sql
-- In V001__memory_entry.sql (generated alongside existing columns):
CREATE TABLE IF NOT EXISTS memory_entry (
    id          VARCHAR(36)  NOT NULL PRIMARY KEY,
    version     BIGINT       NOT NULL,
    session     VARCHAR(36),
    memory_type VARCHAR(255),
    content     TEXT,
    embedding   vector(1536)     -- generated by @Embedding
);

CREATE INDEX IF NOT EXISTS idx_memory_entry_embedding
    ON memory_entry USING ivfflat (embedding vector_cosine_ops)
    WITH (lists = 100);           -- generated by @Embedding
```

The `CREATE EXTENSION IF NOT EXISTS vector` statement is generated once per
module into a dedicated `V000__vector_extension.sql` file (lower version number
to ensure it runs before any aggregate migration).

#### 2. Spring Data JDBC type converter registration

`prefab-vector` auto-configures two Spring `@Component` converters:

- `FloatArrayToVectorConverter` — converts `float[]` to `PGvector` for writes
- `VectorToFloatArrayConverter` — converts `PGvector` to `float[]` for reads

These register themselves via `VectorConverterConfiguration` auto-configuration,
so no user action is required.

#### 3. Generated `@RepositoryMixin` (added to the aggregate's repository)

The processor generates a mixin interface with typed similarity query methods
and merges it into the aggregate's repository — the user does not need to write
a `@RepositoryMixin` manually:

```java
// Generated — merged into MemoryEntryRepository:
public interface MemoryEntrySimilarityMixin {

    /**
     * Finds the {@code limit} MemoryEntry records whose embedding is most
     * similar to {@code queryVector} by cosine distance.
     */
    @Query("""
            SELECT * FROM memory_entry
            ORDER BY embedding <=> :queryVector::vector
            LIMIT :limit
            """)
    List<MemoryEntry> findBySimilarityTo(
            @Param("queryVector") float[] queryVector,
            @Param("limit") int limit);

    /**
     * Finds MemoryEntry records similar to {@code queryVector} filtered
     * by a specific session.
     */
    @Query("""
            SELECT * FROM memory_entry
            WHERE session = :session
            ORDER BY embedding <=> :queryVector::vector
            LIMIT :limit
            """)
    List<MemoryEntry> findBySimilarityToInSession(
            @Param("queryVector") float[] queryVector,
            @Param("session") String session,
            @Param("limit") int limit);
}
```

If the aggregate has `@Filter`-annotated fields, a filtered variant is also
generated for each filter combination.

#### 4. `EmbeddingExcludedFromResponse` behaviour

The embedding field is **excluded from the generated `{Aggregate}Response`
record** by default. Raw embedding vectors are large (6 KB for 1536 float32
values) and meaningless to API consumers. The `@Embedding` annotation
implicitly acts as `@JsonIgnore` on the response DTO.

To override and include the embedding in responses, set
`@Embedding(includeInResponse = true)`.

#### 5. OpenAPI documentation

The `prefab-openapi` plugin is updated to document the similarity query
parameters on `@GetList` endpoints when an `@Embedding` field is present,
noting that similarity search requires a query embedding (not a text string)
and suggesting the application pre-compute embeddings before calling the API.

## Implementation Notes

- The `pgvector` JDBC driver (`com.pgvector:pgvector`) must be declared as an
  optional Maven dependency in `prefab-vector` so callers that use a different
  vector extension (e.g. `Timescale pgvectorscale`) can swap it out.
- The `::vector` cast in the `@Query` SQL is required because Spring Data JDBC
  passes the query vector parameter as a `PGvector` object, and PostgreSQL needs
  an explicit cast to resolve the `<=>` operator for bound parameters.
- For `dimensions` validation: the processor should emit a **warning** (not an
  error) if `dimensions` does not match one of the well-known model output sizes
  (384, 768, 1024, 1536, 3072). Unknown dimensions are valid (custom models exist).
- The `V000__vector_extension.sql` migration version number `000` is chosen to
  precede all aggregate migrations (`V001__`, `V002__`, ...). If a project
  already has a `V000` migration, the processor should emit an error with a
  clear message explaining the conflict.
- Consider adding a `@VectorSearchParam` annotation or `SimilarityQuery` request
  parameter to `@GetList` in a follow-up task so that similarity search can be
  exposed directly on the list endpoint, complementing `@Filter`.

## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
Prefab already generates EventSchemaFactory classes that build org.apache.avro.Schema objects programmatically at compile time. However, the schemas exist only as Java objects at runtime and are never written to disk as .avsc files.

AVSC (Avro Schema) files are standard JSON artefacts that describe an Avro schema. Generating them from the annotated events unlocks several practical benefits:

- Cross-language interoperability: non-Java consumers (Python, Go, etc.) can deserialise the same events without duplicating schema definitions
- Schema evolution tracking: AVSC files can be committed to source control, making schema changes visible in code review
- Schema registry integration: schemas can be pre-registered in a Confluent / AWS Glue / Apicurio registry as part of the build pipeline rather than relying on auto-registration at runtime
- Toolchain compatibility: the Avro Maven plugin and other build tools consume .avsc files directly, enabling code generation for consumers in the same build

The new writer fits naturally inside the existing AvroPlugin pipeline and follows the same compile-time-generation pattern used by every other Prefab plugin.

## Acceptance Criteria
<!-- AC:BEGIN -->
- [ ] #1 A new AvscFileWriter class is added to annotation-processor under the be.appify.prefab.processor.event.avro package; it takes a TypeManifest and writes the resolved Avro schema as a pretty-printed .avsc JSON file
- [ ] #2 Each top-level event type annotated with @Event(serialization = Event.Serialization.AVRO) produces one .avsc file named after the event (e.g. SaleEvents.avsc)
- [ ] #3 Sealed-interface event hierarchies are written as an Avro union schema containing all permitted subtypes, matching the schema already generated by EventSchemaFactoryWriter
- [ ] #4 Nested record types that already generate a dedicated SchemaFactory (i.e. those returned by AvroPlugin.nestedTypes()) are embedded inline in the parent schema and also emitted as individual .avsc files so consumers can reference them independently
- [ ] #5 AVSC files are written to META-INF/avro/ inside the annotation-processor output directory (alongside compiled classes) so they are automatically included in the application JAR and can be loaded from the classpath
- [ ] #6 The content of every generated .avsc file is identical to calling schema.toString(true) on the schema produced by the corresponding EventSchemaFactory at runtime (validated by a test)
- [ ] #7 Unit tests follow the existing pattern in EventSchemaFactoryWriterTest: a compile-testing harness compiles an annotated source fixture and asserts the generated AVSC content matches a stored expected file
- [ ] #8 The AvroPlugin.writeAdditionalFiles() method is updated to call AvscFileWriter for each event and nested type, with no changes required to the @Event annotation or its consumers

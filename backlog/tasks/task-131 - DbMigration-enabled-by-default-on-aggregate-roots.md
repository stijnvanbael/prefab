## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
Currently, developers must explicitly annotate each aggregate root with `@DbMigration` to get Flyway migration scripts
generated. This is an opt-in behaviour that is easy to forget, leading to missing migration scripts and runtime errors.
Since Prefab is opinionated and targets PostgreSQL as the primary relational backend, database migrations should be
generated by default for all aggregate roots when the `prefab-postgres` module is on the classpath.

The `@DbMigration` annotation should gain an `enabled` attribute so developers can still opt out of migration
generation for a specific aggregate when needed.
<!-- SECTION:DESCRIPTION:END -->

## Acceptance Criteria
<!-- AC:BEGIN -->
- [x] #1 Migration scripts are generated for all aggregate roots by default when `prefab-postgres` is on the classpath, without requiring an explicit `@DbMigration` annotation
- [x] #2 `@DbMigration` gains an `enabled` attribute (default `true`) that allows opting out per aggregate (`@DbMigration(enabled = false)`)
- [x] #3 Existing aggregates that already carry `@DbMigration` continue to work without any changes
- [x] #4 Tests cover the default-on behaviour and the explicit opt-out via `enabled = false`
- [x] #5 Documentation is updated to reflect the new default behaviour and the `enabled` attribute
<!-- AC:END -->

## Implementation Plan

<!-- SECTION:PLAN:BEGIN -->
1. Add `enabled` attribute to `@DbMigration` annotation with default value `true`
2. Update the annotation processor to treat every `@Aggregate` as if `@DbMigration` is present unless `@DbMigration(enabled = false)` is set
3. Update `DbMigrationWriter` (or equivalent) to check for the opt-out flag and skip generation when `enabled = false`
4. Add/update tests for default-on generation and explicit opt-out
5. Update `readme.md` to document the new default and the `enabled` attribute
<!-- SECTION:PLAN:END -->

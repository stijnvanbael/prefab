## Description

<!-- SECTION:DESCRIPTION:BEGIN -->
## Context

TASK-227 introduces project-wide plugin configuration via compiler options (e.g. `-Aprefab.plugin.asyncapi.enabled=false`). However, there are use cases where teams need **fine-grained, per-class control**:

1. **Selective opt-out**: "Disable AsyncAPI generation for this aggregate because we don't want to publish events yet"
2. **Alternate output locations**: "Generate this builder as `@TestVisible` in the `test` output instead of the main source"
3. **Mixed-mode generation**: "In the same project, aggregate A sends to Kafka (default), but aggregate B uses custom logic and should skip DbMigration"

Currently, there is no way to override plugin behaviour on a per-aggregate basis without modifying project-wide settings.

## Goal

Introduce a repeatable `@Generate` annotation that allows developers to:

1. **Enable or disable plugins on a per-aggregate basis** — override project defaults for specific classes
2. **Control generated code placement** — specify whether a plugin's output goes to `DEFAULT` (current behavior), `MAIN`, or `TEST` source trees
3. **Compose multiple overrides** — use `@Generate` repeatably to override multiple plugins in a single class

### Proposed API

```java
package io.prefab.core;

/**
 * Per-aggregate plugin override. Repeatable; allows fine-grained control 
 * over code generation behaviour for a single Prefab aggregate.
 * 
 * Overrides project-wide settings from PrefabConfiguration (TASK-227).
 */
@Target({ElementType.TYPE})
@Retention(RetentionPolicy.SOURCE)
@Repeatable(GenerateOverrides.class)
public @interface Generate {
    
    /**
     * The plugin class to configure (e.g. {@code KafkaPlugin.class},
     * {@code AsyncApiDocumentationPlugin.class}).
     */
    Class<?> plugin();
    
    /**
     * Enable or disable this plugin for this aggregate.
     * {@code true} = use default behavior
     * {@code false} = skip all code generation hooks from this plugin
     * 
     * Default: {@code true}
     */
    boolean enabled() default true;
    
    /**
     * Where to emit generated code from this plugin.
     * {@code DEFAULT} = use plugin's default behavior (usually MAIN)
     * {@code MAIN} = emit to src/main/java
     * {@code TEST} = emit to src/test/java (or equivalent)
     * 
     * Default: {@code DEFAULT}
     */
    OutputTarget target() default OutputTarget.DEFAULT;
}

/**
 * Container for multiple {@link Generate} overrides.
 * Generated by compiler when {@link Generate} is used repeatably.
 */
@Target({ElementType.TYPE})
@Retention(RetentionPolicy.SOURCE)
public @interface GenerateOverrides {
    Generate[] value();
}

/**
 * Enum for {@link Generate#target()}.
 */
public enum OutputTarget {
    /** Use plugin's default behavioural setting */
    DEFAULT,
    
    /** Emit to src/main/java (or equivalent in Gradle/Bazel) */
    MAIN,
    
    /** Emit to src/test/java (or equivalent in Gradle/Bazel) */
    TEST
}
```

### Usage Examples

```java
// Example 1: Disable AsyncAPI for this aggregate; use test output for builders
@Generate(plugin = AsyncApiDocumentationPlugin.class, enabled = false)
@Generate(plugin = BuilderPlugin.class, target = OutputTarget.TEST)
@Aggregate(root = "Order")
public class OrderAggregate {
    // ...
}

// Example 2: Force Kafka generation even if disabled globally
@Generate(plugin = KafkaPlugin.class, enabled = true)
@Aggregate(root = "Event")
public class EventAggregate {
    // ...
}

// Example 3: Custom migration scripts in test output
@Generate(plugin = DbMigrationPlugin.class, target = OutputTarget.TEST)
@Aggregate(root = "Account")
public class AccountAggregate {
    // ...
}
```

## Design Requirements

1. **Annotation Definition**: Add `@Generate`, `@GenerateOverrides`, and `OutputTarget` to the `core` module (`be.appify.prefab.core.annotations` package).

2. **Processing in PrefabContext**: 
   - Read `@Generate` overrides from `TypeElement` at aggregate discovery time
   - Parse plugin class and target from annotation
   - Store overrides in a typed structure (e.g. `Map<Class<? extends PrefabPlugin>, PluginOverride>`)

3. **GenerationContext Enhancements**:
   - Extend `GenerationContext` with an `PluginOverrideRegistry` or similar
   - Provide query methods: `isPluginEnabled(Class<? extends PrefabPlugin>)`, `getOutputTarget(Class<? extends PrefabPlugin>)`
   - Fallback to project-wide config (from TASK-227) if no per-aggregate override exists

4. **Plugin Integration**:
   - Each plugin checks `context.isPluginEnabled(this.getClass())` early in its main hook (e.g. `beforeBuild()`)
   - If disabled, set a flag and skip all code generation in that hook
   - Optionally: plugins also respect `context.getOutputTarget(this.getClass())` to emit to `TEST` instead of `MAIN`

5. **Documentation**:
   - Update `backlog/docs/annotation-reference.md` to document `@Generate` with examples
   - Update `backlog/docs/built-in-types.md` to document `OutputTarget` enum
   - Add a section in `backlog/docs/feature-guides.md`: "Per-aggregate plugin overrides"

6. **Testing**:
   - Unit tests verify that `@Generate(plugin = X.class, enabled = false)` causes plugin X to be skipped
   - Unit tests verify that `@Generate(plugin = X.class, target = TEST)` routes output to test source
   - Unit tests verify that multiple `@Generate` annotations on one class are parsed correctly
   - Ensure overrides take precedence over project-wide settings

## Additional Considerations

### Precedence

Per-aggregate overrides should take **strict precedence** over project-wide settings:
```
@Generate(plugin = X, enabled = false)     ← Highest priority
↓
-Aprefab.plugin.X.enabled=true             ← Project-wide default
↓
Default plugin behavior (enabled)           ← Lowest priority
```

### Invalid Plugin Classes

If a developer mistakenly passes a class that is not a `PrefabPlugin`:
- Log a warning during annotation processing
- Ignore the override
- Continue processing

### Future: Plugin-Specific Parameters

The annotation could be extended in the future to support plugin-specific config:
```java
@interface Generate {
    Class<?> plugin();
    boolean enabled() default true;
    OutputTarget target() default OutputTarget.DEFAULT;
    
    // Future:
    // String[] config() default {}; // "key=value" pairs, plugin-dependent
}
```

Currently out of scope, but document the extensibility.

## Implementation Notes

<!-- SECTION:NOTES:BEGIN -->
### Validation & Fail-Fast Strategy

Validation should occur during annotation processing in `PrefabContext` when reading `@Generate` annotations:

1. **Plugin Class Validation**:
   - Verify that `@Generate.plugin()` is assignable from `PrefabPlugin`
   - If not: `throw new ProcessingException("@Generate(plugin=...) must reference a PrefabPlugin subclass, but got: " + pluginClass.getQualifiedName())`
   - Use `processingEnv.getElementUtils().getTypeElement()` and `types.isAssignable()` to check

2. **Plugin Availability**:
   - Check if the plugin class exists on the annotation-processor classpath
   - If the class reference cannot be resolved: raise a compile error
   - Error message: "@Generate references plugin X, but it is not on the annotation-processor classpath"

3. **OutputTarget Compatibility**:
   - Some plugins may only support `DEFAULT` or `MAIN` (e.g., a plugin that modifies `.class` files)
   - If a plugin declares `@SupportsTestOutput` (or similar marker), allow `OutputTarget.TEST`
   - If `OutputTarget.TEST` is used on an unsupported plugin: raise a compile error
   - Future: plugins can declare support; for now, assume all support it and document per-plugin

4. **Aggregate Validation** (optional warning):
   - If `@Generate` is placed on a non-`@Aggregate` class, log a `Note` or `Warning`
   - Message: "@Generate on class X is ignored because it is not an @Aggregate"

5. **Duplicate Overrides** (optional):
   - If the same plugin is configured multiple times (two `@Generate` annotations with `plugin=X`), log a warning
   - First one wins; subsequent ones are ignored

### Error Messages

All validation errors should:
- Include the fully qualified class/annotation name
- Suggest the correct fix (e.g., "Change plugin=X to plugin=io.prefab.kafka.KafkaPlugin.class")
- Be emitted via `processingEnv.getMessager().printMessage(Diagnostic.Kind.ERROR, message, element, annotation)`
